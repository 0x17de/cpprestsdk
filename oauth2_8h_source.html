173 /// <summary>
174 /// OAuth 2.0 configuration.
175 ///
176 /// Encapsulates functionality for:
177 /// - Authenticating requests with an access token.
178 /// - Performing the OAuth 2.0 authorization code grant authorization flow.
179 /// See: http://tools.ietf.org/html/rfc6749#section-4.1
180 /// - Performing the OAuth 2.0 implicit grant authorization flow.
181 /// See: http://tools.ietf.org/html/rfc6749#section-4.2
182 ///
183 /// Performing OAuth 2.0 authorization:
184 /// 1. Set service and client/app parameters:
185 /// - Client/app key & secret (as provided by the service).
186 /// - The service authorization endpoint and token endpoint.
187 /// - Your client/app redirect URI.
188 /// - Use set_state() to assign a unique state string for the authorization
189 /// session (default: "").
190 /// - If needed, use set_bearer_auth() to control bearer token passing in either
191 /// query or header (default: header). See: http://tools.ietf.org/html/rfc6750#section-2
192 /// - If needed, use set_access_token_key() to set "non-standard" access token
193 /// key (default: "access_token").
194 /// - If needed, use set_implicit_grant() to enable implicit grant flow.
195 /// 2. Build authorization URI with build_authorization_uri() and open this in web browser/control.
196 /// 3. The resource owner should then clicks "Yes" to authorize your client/app, and
197 /// as a result the web browser/control is redirected to redirect_uri().
198 /// 5. Capture the redirected URI either in web control or by HTTP listener.
199 /// 6. Pass the redirected URI to token_from_redirected_uri() to obtain access token.
200 /// - The method ensures redirected URI contains same state() as set in step 1.
201 /// - In implicit_grant() is false, this will create HTTP request to fetch access token
202 /// from the service. Otherwise access token is already included in the redirected URI.
203 ///
204 /// Usage for issuing authenticated requests:
205 /// 1. Perform authorization as above to obtain the access token or use an existing token.
206 /// - Some services provide option to generate access tokens for testing purposes.
207 /// 2. Pass the resulting oauth2_config with the access token to http_client_config::set_oauth2().
208 /// 3. Construct http_client with this http_client_config. As a result, all HTTP requests
209 /// by that client will be OAuth 2.0 authenticated.
210 ///
211 /// </summary>
212 class oauth2_config
213 {
214 public:

243  /// <summary>
244  /// Fetch an access token (and possibly a refresh token) based on redirected URI.
245  /// Behavior depends on the implicit_grant() setting.
246  /// If implicit_grant() is false, the URI is parsed for 'code'
247  /// parameter, and then token_from_code() is called with this code.
248  /// See: http://tools.ietf.org/html/rfc6749#section-4.1
249  /// Otherwise, redirect URI fragment part is parsed for 'access_token'
250  /// parameter, which directly contains the token(s).

252  /// In both cases, the 'state' parameter is parsed and is verified to match state().
253  /// </summary>

258  /// <summary>
259  /// Fetches an access token (and possibly a refresh token) from the token endpoint.
260  /// The task creates an HTTP request to the token_endpoint() which exchanges
261  /// the authorization code for the token(s).
262  /// This also sets the refresh token if one was returned.
263  /// See: http://tools.ietf.org/html/rfc6749#section-4.1.3
264  /// </summary>
265  /// <param name="authorization_code">Code received via redirect upon successful authorization.</param>
266  /// <returns>Task that fetches token(s) based on the authorization code.</returns>
267  pplx::task<void> token_from_code(utility::string_t authorization_code)
268  {
269  uri_builder ub;
270  ub.append_query(details::oauth2_strings::grant_type, details::oauth2_strings::authorization_code, false);
271  ub.append_query(details::oauth2_strings::code, uri::encode_data_string(std::move(authorization_code)), false);
272  ub.append_query(details::oauth2_strings::redirect_uri, uri::encode_data_string(redirect_uri()), false);
273  return _request_token(ub);
274  }

276  /// <summary>
277  /// Fetches a new access token (and possibly a new refresh token) using the refresh token.
278  /// The task creates a HTTP request to the token_endpoint().
279  /// If successful, resulting access token is set as active via set_token().
280  /// See: http://tools.ietf.org/html/rfc6749#section-6
281  /// This also sets a new refresh token if one was returned.
282  /// </summary>
283  /// <returns>Task that fetches the token(s) using the refresh token.</returns>
